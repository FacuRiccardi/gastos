### ORM & Driver

- **ORM:** Drizzle ORM (`drizzle-orm`)
- **Driver:** `postgres` (the `postgres.js` driver — Drizzle's recommended driver for PostgreSQL)
- **Migration tool:** `drizzle-kit` — generates SQL migration files from schema changes

### Schema File Location

`src/infrastructure/db/schema/` — one file per bounded context:
- `identity.ts` → households, household_members, users
- `catalogue.ts` → groups, categories
- `expense.ts` → payment_instruments, expenses
- `budget.ts` → budget_limits

`drizzle.config.ts` (at project root) uses `schema: './src/infrastructure/db/schema/*.ts'` to pick up all files.

### Migration Strategy

`drizzle-kit push` (schema push) during development; `drizzle-kit generate` + `drizzle-kit migrate` for production migrations.
Migration SQL files are stored in `src/infrastructure/db/migrations/`.

### Connection / Client Setup

- Shared client: `src/infrastructure/db/client.ts` exports `db` (a Drizzle instance)
- Production env var: `DATABASE_URL`
- Test env var: `TEST_DATABASE_URL`
- Test client: `tests/integration/helpers/testDb.ts` exports `testDb` using `TEST_DATABASE_URL`

### Aggregate-to-Table Mapping Conventions

- **One table per aggregate root.** No table merging or splitting.
- **Column names:** snake_case
- **ID columns:** UUID (native PostgreSQL UUID type — more compact and type-safe than TEXT)
- **Soft-delete:** `deleted_at TIMESTAMPTZ NULL` — NULL means active
- **Money:** two inline columns — `money_amount NUMERIC` + `money_currency TEXT`
- **PaymentMethod:** two inline columns — `payment_method_kind TEXT` (Cash/CreditCard/BankAccount) + `payment_method_instrument_id TEXT NULL`
- **BudgetPeriod:** three inline columns — `period_kind TEXT` (Monthly/Rolling30Days/Custom) + `period_start_date TIMESTAMPTZ NULL` + `period_end_date TIMESTAMPTZ NULL`
- **Household.members:** stored in a `household_members` join table (`household_id TEXT`, `user_id TEXT`, composite PK). Single source of truth for the relationship. `users` has NO `household_id` column — `UserRepository` reconstructs `User.householdId` by querying `household_members WHERE user_id = ?`.
- **Audit columns:** every table has `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()` and `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`. `updated_at` is set to `NOW()` on every upsert.

### Integration Test Setup

- **Isolation strategy:** Truncate before each test (`TRUNCATE ... RESTART IDENTITY CASCADE` in `beforeEach`)
- **Test helpers:**
  - `tests/integration/helpers/testDb.ts` — exports `testDb` (Drizzle instance) and `clearTables()`
  - `tests/integration/helpers/repositories.ts` — exports `makeRepos()`, a factory that returns all 7 Drizzle repositories wired to `testDb`
- **Database:** pointed at `TEST_DATABASE_URL` — never uses `DATABASE_URL`
- **Parallelism:** `fileParallelism: false` in `vitest.config.integration.ts` — test files run serially since they share the same database

### Integration Test Layers

Two sub-trees under `tests/integration/`, each covering a different concern:

**`tests/integration/repository/`** — verifies that each Drizzle repository correctly persists and queries data against PostgreSQL. Covers: field round-trips, upsert behavior, soft-delete filters, pagination, type reconstruction (branded IDs, Value Objects, enums), and hard-delete. One file per repository, 56 tests total.

**`tests/integration/application/`** — verifies that each use case works end-to-end with real Drizzle repositories and a real database. Covers: data persisted by one use case is retrievable by another, error cases throw the right messages, and complex orchestration (e.g. `GetBudgetLimitBalance` aggregating expenses across a group's categories) produces correct results. One file per bounded context, 37 tests total.

Run with: `devbox run test:integration` (requires `devbox services up` to have PostgreSQL running).

## Layer Overview

```
src/
  domain/         ← pure business logic (this document)
  application/    ← use case orchestration (calls domain, drives ports)
  infrastructure/ ← repository implementations, DB, external services
  http/           ← HTTP adapters (Fastify routes/controllers)

tests/
  unit/
    domain/       ← mirrors src/domain/ folder-for-folder
    application/  ← mirrors src/application/ folder-for-folder
  helpers/        ← in-memory repository implementations shared across application tests
  integration/    ← tests that span infrastructure adapters
```

The domain layer has **zero** dependencies on application, infrastructure, or http. It knows nothing about databases, HTTP, or frameworks.

### Use Case Shape

Every use case follows this pattern:

```typescript
export interface LogExpenseInput {
  householdId: HouseholdId;
  userId: UserId;
  // ... all required inputs using domain types
}

export type LogExpenseOutput = { id: ExpenseId };

export class LogExpense {
  constructor(
    private readonly expenses: ExpenseRepository,
    private readonly categories: CategoryRepository,
  ) {}

  async execute(input: LogExpenseInput): Promise<LogExpenseOutput> {
    // 1. load / validate via repositories
    // 2. call domain methods
    // 3. persist via repositories
    // 4. return output
  }
}
```

**Rules:**
- Input and output types use domain types (branded IDs, VOs) — no raw strings
- The constructor declares only the repository interfaces it actually needs
- `execute()` is always `async` and returns a `Promise`
- Use cases do not call other use cases — they call repositories and domain methods directly
- Cross-context queries (e.g. resolving GroupId → CategoryIds for ListExpenses) are orchestrated here

## Design Patterns Used in the Domain

### Value Object
**Where:** every file not named an Aggregate Root or Repository.
**Rule:** immutable; equality is by value, not reference; no identity field.
**Examples:** `Money`, `Currency`, `PaymentMethod`, `InstallmentPlan`, `ExpenseDate`, `BudgetPeriod`, all `*Id` types.

### Repository (Port)
**Where:** one `*Repository.ts` interface per Aggregate Root, inside the aggregate's folder.
**Rule:** the interface lives in the domain; the implementation lives in `infrastructure/`. The domain only ever depends on the interface.
**Examples:** `HouseholdRepository`, `ExpenseRepository`, `BudgetLimitRepository`.

### Static Factory Method
**Where:** `BudgetLimit.ts`.
**Why:** `BudgetLimit` must target either a `CategoryId` or a `GroupId` — never both, never neither. Constructing it directly would require external validation. Two static factories make each valid state explicit and unconstruable the wrong way.
```
BudgetLimit.forCategory(id, householdId, money, period)
BudgetLimit.forGroup(id, householdId, money, period)
```

## Conventions

- **Aggregate Root (AR):** the only entry point into its consistency boundary. External code holds a reference to the AR, never to internal entities directly.
- **Immutability:** Value Objects are constructed once and never mutated. Methods that "change" a VO return a new instance.
- **No infrastructure imports:** nothing inside `src/domain/` may import from `src/application/`, `src/infrastructure/`, or `src/http/`.
- **Errors:** each context defines its own domain error classes colocated with the aggregate (e.g., `catalogue/group/GroupNotFoundError.ts`). No generic error wrappers.
- **IDs:** every `*Id` is a branded type wrapping a `string` UUID. Branding makes them nominally distinct so TypeScript rejects a `UserId` where a `HouseholdId` is expected. Each `*Id` file also exposes a `generate()` static method that calls `crypto.randomUUID()` internally. IDs are never bare primitives in domain signatures.
